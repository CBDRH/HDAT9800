This tutorial discusses the RShiny application framework.

Shiny

Shiny apps consist of a user-interface running in a user’s browser and a server application with which it communicates.

The Shiny user interface (UI)

Inputs in the UI are controlled by the user. Their values are sent to the server which generates outputs. The outputs are sent back to the UI. The UI displays these outputs.

For example, here is a simple application consisting of a UI with one input and one output:

  • the input is a selection menu (drop down) containing a list of regions
  • the output is a plot

In this example, the input is called region and the output is called phonePlot. You can see this in the example below

selectInput("region", "Region:", choices = colnames(WorldPhones))
plotOutput("phonePlot")

Try changing the input dropdown and you will see the plot changes appropriately.

The Shiny server

The UI is communicating with a server running this code

output$phonePlot = renderPlot({
  barplot(WorldPhones[,input$region] * 1000,
          ylab = "Number of Telephones", xlab = "Year")
})

This is a very simple example.

This code says “create a barplot using the region input, render it for transmission and display and send it to the UI to be displayed in the output named phonePlot”.

The UI

The UI in our example consists of just two elements

  • a drop down input
  • a graph

A Shiny UI consists of a sequence of commands which builds up the HTML necessary to create the interface.

Shiny provides convenience functions to creates these input elements. The user changes these to communicate information to the Shiny application. They might be text input fields, menus, check boxes, radio buttons, anything that HTML provides to accept user input.

A lot needs to go on under the bonnet to make this work interactively and dynamically but thankfully that’s precisely the detail that Shiny hides from us and does for us.

The graph

Just as the selectInput() function created the input element, the plotOutput() function creates HTML to display information coming from the server for display to the user.

There are different output functions for different types of R output. Here the code

plotOutput("phonePlot")

says

  • generate HTML to display a plot
  • use the information the server assigns to the output called phonePlot as the plot to display.

When we look at the server code we’ll see the server generating a plot, processing it into a form which can be sent to the UI and then assigning it to the phonePlot output.

The Server

The code

  barplot(WorldPhones[,input$region] * 1000,
          ylab = "Number of Telephones", xlab = "Year")

creates a bar plot using the WorldPhones dataset.

This is wrapped in a function renderPlot() which takes a plot and does everything necessary to make sure the UI knows about it and can display it.

renderPlot({
  barplot(WorldPhones[,input$region] * 1000,
          ylab = "Number of Telephones", xlab = "Year")
})

Note that the argument to renderPlot() is actually a closure (a block of code surround by {...} like the body of a function). Remember, a closure is code which is not executed immediately but which is passed around for later execution. renderPlot() (and indeed all the other render-type functions) takes a closure because it will need to recalculate the plot whenever an input changes.

Finally, the output of renderPlot() is assigned to output$phonePlot. The name phonePlot corresponds to the name used in the UI file in plotOutput().

output$phonePlot = renderPlot({
  barplot(WorldPhones[,input$region] * 1000,
          ylab = "Number of Telephones", xlab = "Year")
})

When does Shiny recalculate the output plot?

Shiny looks at the closure supplied to renderPlot and works out what inputs it depends on. In this case the only input it depends on is input$region.

Shiny tries to keep recalculation to a minimum. It will only recalculate output$phonePlot when it needs to, in this case, when input$region changes.

Structuring Shiny applications

Historically a Shiny app was broken into two files:

  • ui.R — containing the UI code
  • server.R — containing the server code

These days a Shiny application can be contained in a single file

  • app.R — containing UI and server code

Whichever approach you use, these filenames must be used exactly.

Note that, these files need not contain every scrap of code comprising your application. These files contain the top level definitions of the UI and the server. Code can be included from other files and normal software best practice can still be followed:

  • keeping related content in one file
  • separating unrelated content into separate files
  • breaking code up into separate files to make things more manageable

The dual file approach

In the dual file approach we have two separate files. One file ui.R contains the UI definition. Another files server.R contains the server definition.

The UI file: ui.R

In this file we create the UI.

# ui.R

library(shiny)

fluidPage(
  sliderInput(inputId = "num",
    label = "Choose a number",
    value = 25,
    min = 1, max = 100),
  plotOutput("hist")
)

Previously saw just the tag functions to create elements of the Shiny UI. Here we are creating an entire page. The fluidPage() function takes a variable number of arguments and creates an HTML page. The fluidPage() function takes an optional title= argument to set the page title. See the documentation for other arguments.

Under the bonnet…

Let’s look at the HTML code generated by fluidPage().

Here we’ve set the title and used simple elements, e.g. p() and em(), for illustrative purposes.

fluidPage(
  p("Paragraph 1"),
  p("Paragraph 2 contains", em("some emphasised text")),
  p("Trailing paragraph")
)

which gives

<div class="container-fluid">
  <p>Paragraph 1</p>
  <p>
    Paragraph 2 contains
    <em>some emphasised text</em>
  </p>
  <p>Trailing paragraph</p>
</div>

As you can see, it just concatenates the given elements.

The server file: server.R

The server.R file contains the definition of the server’s input/output function:

# server.R

library(shiny)

function(input, output) {
  output$hist <- renderPlot({
    hist(rnorm(input$num))
  })
}

This function uses the input called num to generate the content of an output called hist.

Shiny apps do sophisticated things under the hood to work out what has changed and what needs to be regenerated when inputs change. Shiny apps do not simply recalculate everything every time something changes.

Shiny give us quite a bit of control over this reactivity of outputs but those details are for another time.

The single file approach

In a single file approach the UI and the server are defined in the same file and the app instantiated from those two separate components.

library(shiny)

ui <- fluidPage(
  sliderInput(inputId = "num",
    label = "Choose a number",
    value = 25,
    min = 1, max = 100),
  plotOutput("hist")
)

server <- function(input, output) {
  output$hist <- renderPlot({
    hist(rnorm(input$num))
  })
}

shinyApp(ui = ui, server = server)

Aside…

Of course, as mentioned, there is no need to have all the code in a single file.

The single file approach just means that the application is built in this single file called app.R.

Components of the application can be defined elsewhere and referred to in this file.

library(shiny)

ui <- buildMyUi(...)
server <- buildMyServer(...)

shinyApp(ui = ui, server = server)

Running Shiny apps

Shiny applications are run by calling runApp() with the name of the directory containing the app.R (or ui.R and server.R) files.

runApp("myApp")

RStudio recognises Shiny applications and provides a ▶ Run App button at the top of the window to run the application.

Close the window or press the stop button 🛑 to stop the app

Nicer user interfaces

The interfaces we have been constructing so far have been simple affairs with a primitive layout consisting of sequential interface elements.

Shiny gives us utility functions to create more interesting (and better) interfaces.

In particular, the titlePanel(), sidebarLayout(), sidebarPanel() and mainPanel() functions can be used to create an interface with:

  • a title across the top

Output elements

Output elements tell Shiny how to display the output in the application.

The code to tell Shiny how to build the output lives in the server.

  • dataTableOutput
  • htmlOutput
  • imageOutput
  • plotOutput
  • tableOutput
  • textOutput
  • uiOutput
  • verbatimTextOutput

These elements broadly correspond to the type of the data being displayed.

Some packages extend this list. For example, leaflet provides output elements to display interactive maps.

There are two parts to output:

  • the UI display part
  • the server calculation part

The server must send what the UI expects, i.e. they should match.

ui <- fluidPage(
  sliderInput(inputId = "num",
    label = "Choose a number",
    value = 25,
    min = 1, max = 100),
  plotOutput("hist")                  <--- plotOutput
)

server <- function(input, output) {
  output$hist <- renderPlot({         <--- renderPlot
    hist(rnorm(input$num))
  })
}

Output server code chunks contain code to render output in a format suitable for the UI elements:

  • renderDataTable
  • renderImage
  • renderPlot
  • renderPrint
  • renderTable
  • renderText
  • renderUI

Render functions take a closure (code surrounded by {...}) because it’s an anonymous function.

Exactly like a function, Shiny needs this code not to be executed when it is first seen but at some later point (when the input changes).

Naming inputs and outputs

It is important that the names of your inputs in the UI match up with the names that the server function expects in the server.

Similarly, it’s important that the server assigns to outputs with names that match output elements in the UI.

Failure to use matching names will lead to nothing happening when you change input or output getting lost and never displayed.

Case is important.

The name myinput is different to myInput and both are different to MyInput.

What happens when inputs change?

Let’s take a look at what happens when the user changes an input

  • the user changes the input
  • the change is sent from the UI to the server
  • the server determines which outputs depend on the input and recalculates them by running the closure for that output
  • the result(s) is/are sent back to the UI
  • the UI displays any change(s)

It’s important to note:

  • a single input may result in multiple outputs changing
  • a single output may depend on multiple inputs

Consider the following application with the server code

server <- function(input, output) {
  output$checkbox1.status = renderText({
    if (input$checkbox1) {
      "is checked"
    } else {
      "is not checked"
    }
  })
  output$checkbox2.status = renderText({
    if (input$checkbox2) {
      "is checked"
    } else {
      "is not checked"
    }
  })
  output$numberOfCheckboxesChecked = renderText({
    sum(c(input$checkbox1, input$checkbox2))
  })
}

and the UI as follows:

fluidPage(
  checkboxInput("checkbox1", "Checkbox 1"),
  checkboxInput("checkbox2", "Checkbox 2"),
  hr(),
  p("Checkbox 1 ", textOutput("checkbox1.status", inline=TRUE)),
  p("Checkbox 2 ", textOutput("checkbox2.status", inline=TRUE)),
  p("The total number of checkboxes checked is: ", textOutput("numberOfCheckboxesChecked", inline=TRUE))
)

Notice that:

  • changing the input checkbox1 results in the outputs checkbox1.status and numberOfCheckboxesChecked changing
  • changing the input checkbox2 results in the outputs checkbox2.status and numberOfCheckboxesChecked changing

Shiny works out which output must react to which changed inputs.

Aside…

The server function has two arguments input and output which are lists.

When inputs change the updated input object and the output object are passed to the server function. The server function recalculates the output slots as required.

The closures (code between the {...}) passed to the render functions for each output slot are only called when any inputs referred to in the closure are changed.

For example, looking at

output$numberOfCheckboxesChecked = renderText({
  sum(c(input$checkbox1, input$checkbox2))
})

we can see the closure {sum(c(input$checkbox1, input$checkbox2))} which references input$checkbox1 and input$checkbox2. If either of these inputs changed the closure is executed. (The code here relies on the fact that checkbox input values are TRUE or FALSE: it creates a logical vector out of them and sums the number of TRUE values in the vector giving the total number of checked checkboxes.)

If input$checkbox1 is changed the closures associated with output$checkbox1.status and output$numberOfCheckboxesChecked are executed but the closure associated with `output$checkbox2.status is not executed because it does not reference input$checkbox1.

Once the closure is executed the result is prepared for output (here it’s just rendered as plain text) and assigned to output$numberOfCheckboxesChecked. This results in the UI textOutput("numberOfCheckboxesChecked", inline=TRUE) getting updated.

Summary

This concludes our introductory tutorial on Shiny.

Shiny applications consist of two parts:

  • the user interface (UI)
  • the server

These two parts communicate using named input and outputs.
